エンジニアは複雑な技術を扱うが、その複雑さを他者に伝えるとき、しばしば壁にぶつかる。Slackで質問したが意図が伝わらない。PRのdescriptionを書いたが、レビュアーに何度も確認される。ドキュメントを残したが、誰も読まない。

これらの問題の根本にあるのは「Clarity（明瞭さ）」の欠如だ。クラリティフレームワークは、明瞭なコミュニケーションを実現するための思考の枠組みである。

## なぜ明瞭さが重要か

エンジニアリングはチームスポーツだ。一人で完結する仕事はほとんどない。コードレビュー、障害対応、仕様の議論、技術選定。すべてはコミュニケーションの上に成り立っている。

明瞭さが欠けたコミュニケーションは、単に非効率なだけではない。

```
曖昧なメッセージ
  ↓ 解釈の分岐
複数の理解
  ↓ 認識のズレ
手戻り・バグ・対立
```

Slackで「これって大丈夫ですか？」と聞かれても、「これ」が何を指すのか、「大丈夫」が何を意味するのかがわからなければ、答えようがない。明瞭さは、思いやりでもある。

## クラリティフレームワークの5つの要素

明瞭なコミュニケーションを構成する要素を5つに分類する。

### 1. 目的の明示（Purpose）

最初に「なぜこの文章を書いているのか」を明確にする。

**悪い例**：
> データベースの件なんですが、昨日のミーティングで話したやつで、ちょっと確認したいことがあって...

**良い例**：
> 【確認依頼】users テーブルへのインデックス追加について判断をお願いします

目的を明示すると、読み手は「どう反応すべきか」がわかる。情報共有なのか、判断を求めているのか、作業依頼なのか。これが曖昧だと、読み手は何をすればいいかわからず、スレッドが迷走する。

### 2. 構造化（Structure）

情報を論理的な順序で配置する。人間の脳は構造化された情報を処理しやすい。

**基本構造**：
1. 結論・要点（何が言いたいか）
2. 背景・理由（なぜそう言えるか）
3. 詳細・補足（必要な追加情報）
4. 次のアクション（何をしてほしいか）

技術ドキュメントであれば：
1. これは何か（概要）
2. なぜ必要か（動機）
3. どう使うか（使用方法）
4. どう動くか（仕組み）

PREP法（Point-Reason-Example-Point）やSDS法（Summary-Detail-Summary）も、この構造化の具体的なテクニックだ。

### 3. 具体性（Specificity）

抽象的な表現を避け、具体的に書く。

**曖昧**：
> パフォーマンスに問題がある

**具体的**：
> `/api/users` のレスポンスタイムがP95で800msを超えている（SLO目標は200ms）

数値、日時、ファイルパス、エラーメッセージ。具体的な情報があれば、読み手は自分で判断できる。曖昧な表現は、解釈を読み手に委ねてしまう。

### 4. 簡潔性（Conciseness）

必要な情報を過不足なく伝える。長ければ良いわけではない。

**冗長**：
> 先日から検討を重ねてきた結果として、様々な観点から総合的に判断した上で、今回のリリースについては延期する方向で進めたいと考えております。

**簡潔**：
> 今回のリリースは延期します。理由：ステージング環境でメモリリークが見つかったため。

簡潔さのポイント：
- 一文一意（一つの文に一つの情報）
- 受動態より能動態
- 名詞化を避ける（「検討を行う」→「検討する」）
- 冗長な敬語を省く（ビジネスチャットでは特に）

### 5. 文脈の共有（Context）

読み手が持っていない情報を補う。自分には自明でも、相手には自明ではない。

**文脈不足**：
> 例の件、やっぱりダメでした

**文脈あり**：
> 昨日のSREミーティングで議論したCloudflare移行の件について、インフラチームから「現時点では優先度が低い」との回答がありました

リンクを貼る、背景を一文で説明する、関連するIssueやPRを参照する。少しの手間で、相手の認知負荷は大きく下がる。

## 実践：Slackでの適用

Slackは特に明瞭さが求められる。非同期コミュニケーションでは、即座に確認できないからだ。

**質問のテンプレート**：
```
【目的】〇〇について教えてください / 判断をお願いします / 相談させてください
【背景】現在△△の対応をしています
【質問】□□の場合、どうすべきでしょうか？
【自分の考え】私は◇◇だと思っていますが...
```

**報告のテンプレート**：
```
【結論】〇〇が完了しました / 〇〇の問題が発生しています
【詳細】
- 何が起きたか
- 影響範囲
- 対応状況
【次のアクション】△△を予定しています / □□の判断をお願いします
```

## 実践：PRのdescriptionでの適用

レビュアーの時間を尊重するPR descriptionを書く。

```markdown
## 概要
ユーザー検索機能にページネーションを追加

## 背景
- 現状：全件取得しているため、ユーザー数増加に伴いレスポンス遅延が発生
- 対応：カーソルベースのページネーションを実装

## 変更内容
- `UserRepository#search` にページネーションパラメータを追加
- レスポンスに `next_cursor` を含める
- 既存のAPIは後方互換性を維持

## 確認してほしいポイント
- カーソルの実装方法（ULIDを使用）は妥当か
- エッジケース（0件、最終ページ）の処理

## テスト
- 単体テスト追加済み
- ステージング環境で動作確認済み
```

## アンチパターン

明瞭さを損なう典型的なパターン：

**前提の省略**：「あれ、どうなりました？」→ 何の話かわからない
**曖昧な指示代名詞**：「これを直してください」→ 何を直すのか
**暗黙の期待**：「なるはやで」→ いつまでに必要なのか
**保険的な曖昧さ**：「たぶん大丈夫だと思います」→ 大丈夫なのかどうか
**情報の過剰**：全ての調査結果を羅列 → 結論がわからない

## 明瞭さは訓練で身につく

明瞭に書く能力は、生まれつきの才能ではない。意識的な訓練で向上する。

**書く前に考える**：いきなり書き始めず、「何を伝えたいか」を整理する。TOWER法（Think, Organize, Write, Edit, Rewrite）はこのプロセスを定式化したものだ。

**推敲する**：書いたら必ず読み返す。「初めて読む人に伝わるか」の視点でチェックする。

**フィードバックを求める**：「わかりにくい」と言われたら、それは成長の機会だ。どこがわかりにくかったかを聞き、改善する。

**良い例を集める**：明瞭なドキュメントやメッセージを見つけたら、なぜ明瞭なのかを分析する。パターンを抽出して、自分の引き出しにする。

## コードとの共通点

明瞭な文章を書く力は、良いコードを書く力と根底でつながっている。

| 文章 | コード |
|------|--------|
| 目的の明示 | 関数名・コメント |
| 構造化 | モジュール分割 |
| 具体性 | 明確な型・命名 |
| 簡潔性 | DRY・シンプルな実装 |
| 文脈の共有 | ドキュメント・テスト |

良いコードは「読んでわかる」コードだ。良い文章も同じ。読み手の認知負荷を下げ、意図を明確に伝える。その原則は変わらない。

## まとめ

クラリティフレームワークの5要素：

1. **目的の明示**：なぜ書いているかを最初に示す
2. **構造化**：論理的な順序で情報を配置する
3. **具体性**：抽象的な表現を避け、具体的に書く
4. **簡潔性**：過不足なく、必要な情報だけを伝える
5. **文脈の共有**：読み手が持っていない情報を補う

明瞭さは、相手への思いやりだ。自分の頭の中を整理し、相手が理解しやすい形で届ける。その一手間が、チーム全体の生産性を上げる。

次にSlackでメッセージを書くとき、PRを出すとき、ドキュメントを残すとき。5つの要素を意識してみてほしい。
